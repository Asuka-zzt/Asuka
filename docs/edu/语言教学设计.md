# 语言教学功能设计 — 双 persona 语言老师（对话陪练 + 练习生成 + TTS）

> 目标：面向**中文母语者**，提供「英语老师」「日语老师」两个独立 persona 的 AI 陪练。
> 用户与老师自由对话，老师**即时中文纠错 + 给地道说法**；用户可让老师**按水平生成练习/测验**；
> 目标语句子可**一键 TTS 朗读**。
>
> 复用现有四大基建（多 persona Agent / `@tool` 工具 / react agent 循环 / 前端 view），
> **不另起炉灶**。沿用 Phase 3 wiki 工具的 `@tool` + `with_structured_output` 范式。

---

## 1. 范围

本设计经与用户确认，MVP 锁定如下（其余登记 §13 二期 TODO）。

**做（MVP）：**
- 两个独立 persona：**英语老师** / **日语老师**，各自 Soul（中文讲解 + 目标语沉浸 + 针对性纠错 checklist）。
- **对话陪练 + 即时纠错**：对话中老师指出错误、给地道改写。提供结构化 `correct_text` 工具，供前端渲染「纠错卡片」。
- **练习/测验生成**：`generate_quiz` 工具，按语言 + 水平（CEFR / JLPT）+ 题型生成选择/填空/翻译题，含**中文解析**与**音标/假名标注**。
- **TTS 朗读**：`edge-tts`（免费、无需 Key、支持英/日多音色），通过 REST 路由 `/api/tts` 下发音频；前端在目标语句子旁加朗读按钮。
- 工具按 persona 分配：老师 persona 拿语言工具，不拿 wiki 工具（升级 `registry.py`）。

**不做（→ §13 二期 TODO）：**
- 生词卡 + SRS 间隔复习（依赖 `db/` 表）。
- 跨会话「错误记忆」注入（依赖 `core/memory/` long 层，目前未落地）。
- 语法专题课讲解（独立教学课文/课程体系）。
- STT 跟读评分（只做 TTS，不做语音识别）。
- 持久化 `LearnerProfile`（MVP 由前端选水平、随请求参数传入；落库归二期）。
- 自适应出题 / 教学 LangGraph 子图（MVP 单结构化工具足矣，见 §8 权衡）。

### 2.1 复用现有基建

| 现有基建 | 语言教学怎么用 |
|---|---|
| 多 persona Agent | 预置「英语老师」「日语老师」两个 `AgentConfig`，各自 Soul |
| `create_agent` react 循环 | 老师自主决定何时调用 `correct_text` / `generate_quiz`，与 wiki 工具同一机制 |

### 2.2 当前代码现状（诚实标注，避免过度设计）

- **无 `db/` 层、无 `core/memory/` 层**：故 SRS、跨会话错误记忆、持久化画像**全部归二期**，MVP 不被迫先建整套 db/memory。

  → MVP 需做**最小升级**：支持按 persona id 选取 agent 配置与工具（§5、§7、§11）。
- `config.py` 已有 `get_settings()` 单例与 wiki 配置，按同样模式追加 TTS 配置。

## 3. 目标用户与教学策略

用户是**中文母语者**，错误具有**系统性方向**，prompt 工程吃这个红利（写进 Soul 的纠错 checklist）：

| 语言 | 中文母语者高频系统性错误（纠错重点） |
|---|---|
| 英语 | 冠词 a/an/the、时态（尤其完成体/过去 vs 现在）、单复数与可数性、介词搭配、主谓一致 |
| 日语 | 助词 は/が/を/に/で、敬语（です・ます／尊敬・谦让）、自他动词、授受动词（あげる/くれる/もらう）、长短音与促音 |

**标注策略**：日语输出**加假名/振假名 + 罗马音**；英语**加音标（IPA）**。纠错与解析**一律用中文**，降低理解门槛；例句/改写用目标语，保证沉浸。

## 4. 整体架构与数据流

```
前端「学习」页
  选择 persona（english_teacher / japanese_teacher）+ 水平（CEFR / JLPT）
  │
  ├── 聊天 (WS /ws/{conv})
  │     → dispatch.build_agent(persona_cfg)   # 老师 persona，注入语言工具
  │     → react agent：
  │           · 自由对话纠错（Soul 驱动，inline）
  │           · 需结构化纠错卡 → 调 correct_text(text, language)
  │           · 用户要练习 → 调 generate_quiz(language, level, topic, type, n)
  │     ← astream_events 流式回复（含工具结构化结果）
  │
  └── 朗读按钮 (REST POST /api/tts {text, language})
        → api/provider/tts.synth(text, voice)   # edge-tts
        ← audio/mpeg (mp3 bytes) → 前端 <audio> 播放
```

水平（level）MVP 作为**请求参数**：前端选择后随 WS 消息上下文 / 工具参数传入，并注入 persona Soul。**不落库**（二期 `LearnerProfile`）。

## 6. Persona 设计（`core/agent/presets.py`）

两个预置 `AgentConfig`，核心在 Soul。Soul 内写死 §3 的纠错 checklist + 标注规则 + 中文讲解约束。

```python
def english_teacher(level: str = "B1") -> AgentConfig:
    return AgentConfig(
        id="english_teacher",
        name="英语老师 Emily",
        model_id=get_settings().default_model,
        language="english",
        level=level,
        soul=ENGLISH_TEACHER_SOUL.format(level=level),
    )

def japanese_teacher(level: str = "N4") -> AgentConfig:
    return AgentConfig(
        id="japanese_teacher",
        name="日语老师 Asuka",
        language="japanese",
        level=level,
        soul=JAPANESE_TEACHER_SOUL.format(level=level),
    )
```

Soul 模板要点（中文写就，约束行为）：
- 角色：耐心的母语老师，面向中文学习者，**解释一律中文，例句/改写用目标语**。
- 难度对齐 `{level}`（CEFR / JLPT），用词与句长不超纲。
- 纠错风格：先肯定 → 指出错误（点名错误类型，对照 §3 checklist）→ 给地道改写 → 一句话讲清「为什么」。
- 标注：日语句子配假名 + 罗马音；英语配音标。
- 工具触发指引：用户给出一段目标语让你批改 → 用 `correct_text`；用户要练习/测验/题 → 用 `generate_quiz`。
- 沉浸但不失中文脚手架：陪练场景（点餐/面试/问路…）用目标语开场，学习者卡住时用中文提示。

> `AgentConfig` 改动最小：新增 `language: str | None = None`、`level: str | None = None`（可选，向后兼容现有 default agent）。

## 7. 对话陪练 + 即时纠错

**两条路径并存：**

1. **Soul 驱动的 inline 纠错（主路径）**：日常对话中，老师按 Soul 自然地在回复里纠错+改写。无需工具，零额外成本，流式体验最佳。
2. **结构化 `correct_text` 工具（卡片路径）**：当用户明确「帮我改这段」或前端「批改」按钮触发时，react agent 调 `correct_text`，返回结构化结果，前端渲染**纠错卡片**（逐条错误对照 + 地道改写 + 中文讲解）。

```python
# core/tools/language.py
@tool
async def correct_text(text: str, language: str) -> CorrectionResult:
    """批改一段目标语文本，返回逐条纠错、地道改写与中文讲解。
    当用户要求"批改/纠正/检查我写的这句"时调用。language 取 "english" 或 "japanese"。
    """
    llm = get_llm(get_settings().default_model)
    return await llm.with_structured_output(CorrectionResult).ainvoke(
        CORRECT_PROMPT.format(text=text, language=language, checklist=_checklist(language))
    )
```

```python
# core/tools/schemas.py
class CorrectionItem(BaseModel):
    error_type: str          # "冠词" / "时态" / "助词" 等（对齐 §3 checklist）
    original: str            # 原文片段
    corrected: str          # 改正后片段
    explanation_zh: str     # 中文讲解，一句话说清为什么

class CorrectionResult(BaseModel):
    has_error: bool
    items: list[CorrectionItem]
    natural_rewrite: str     # 整句地道改写（目标语）
    annotation: str          # 改写的音标(英) / 假名+罗马音(日)
    encouragement_zh: str    # 一句中文鼓励
```

> 跨会话「记住你常犯的错」依赖 memory long 层 → **二期**。MVP 纠错只在当前会话上下文内（react agent + checkpointer 短期记忆）。

## 8. 练习 / 测验生成（`generate_quiz` 工具）

```python
@tool
async def generate_quiz(
    language: str,
    level: str,
    topic: str = "",
    quiz_type: str = "mcq",   # mcq | cloze | translation
    n: int = 5,
) -> QuizSet:
    """按语言、水平、主题、题型生成 n 道练习题，含中文解析与音标/假名标注。
    当用户要"练习/测验/出题/做题"时调用。language ∈ {english, japanese}；
    level 为 CEFR(A1–C2) 或 JLPT(N5–N1)；quiz_type ∈ {mcq, cloze, translation}。
    """
    llm = get_llm(get_settings().default_model)
    return await llm.with_structured_output(QuizSet).ainvoke(
        QUIZ_PROMPT.format(language=language, level=level, topic=topic or "日常",
                           quiz_type=quiz_type, n=n)
    )
```

```python
class QuizItem(BaseModel):
    type: str                       # mcq | cloze | translation
    question: str                   # 题干（目标语，cloze 用 ___ 留空）
    options: list[str] | None       # mcq 选项；其余为 None
    answer: str                     # 正确答案
    explanation_zh: str             # 中文解析
    annotation: str                 # 音标(英) / 假名+罗马音(日)

class QuizSet(BaseModel):
    language: str
    level: str
    topic: str
    items: list[QuizItem]
```

**为什么是单工具而非 LangGraph 子图？** Phase 3 wiki 用子图是为了 `Send` 并发写章节；出题**无 fan-out 需求**，单次 `with_structured_output` 即可，符合「最小改动 / 框架内置优先」。自适应出题（按错误记忆动态调难度、多步评估）需要 memory 层与多节点编排 → **二期**再引入 `core/graph/lesson/` 子图（见 §13）。

## 9. TTS 朗读（`edge-tts`）

TTS 不需要 LLM 决策，**不做成 `@tool`**，而是独立 REST 路由，前端朗读按钮直接调用（更简单、更快、可缓存）。

```python
# api/provider/tts.py
import edge_tts

async def synth(text: str, language: str) -> bytes:
    """文本 → mp3 字节。language 选音色（en-US / ja-JP）。"""
    voice = get_settings().tts_voices.get(language, "en-US-AriaNeural")
    communicate = edge_tts.Communicate(text, voice, rate=get_settings().tts_rate)
    buf = bytearray()
    async for chunk in communicate.stream():
        if chunk["type"] == "audio":
            buf.extend(chunk["data"])
    return bytes(buf)
```

```python
# routes/tts.py
@router.post("/api/tts")
async def tts(req: TTSRequest) -> Response:
    audio = await synth(req.text, req.language)
    return Response(content=audio, media_type="audio/mpeg")
```

```python
# config.py 追加
tts_voices: dict[str, str] = {
    "english": "en-US-AriaNeural",
    "japanese": "ja-JP-NanamiNeural",
}
tts_rate: str = "-10%"   # 学习场景略放慢
```

依赖：`uv add edge-tts`（纯 Python、调用微软公共端点、无需 API Key）。

当前 `registry` 对所有 agent 返回同一套、`dispatch` 只缓存单 agent。最小升级：

```python
# core/tools/registry.py
_TEACHER_TOOLS = [correct_text, generate_quiz]
_DEFAULT_TOOLS = [generate_wiki]

def get_tools_for_agent(agent_id: str) -> list[BaseTool]:
    if agent_id in {"english_teacher", "japanese_teacher"}:
        return list(_TEACHER_TOOLS)
    return list(_DEFAULT_TOOLS)
```

```python
# core/graph/dispatch.py：单缓存 → 按 id 缓存
_agents: dict[str, Any] = {}

async def build_agent(config: AgentConfig | None = None) -> Any:
    cfg = config or default_agent()
    if cfg.id not in _agents:
        _agents[cfg.id] = create_agent(
            model=get_llm(cfg.model_id),
```

## 13. 关键设计权衡

| 决策 | 选择 | 理由 / 代价 |
|---|---|---|
| 出题机制 | **单 `@tool` + structured_output** | 无 fan-out，子图过重；自适应/多步出题归二期 `core/graph/lesson/` |
| 纠错机制 | **Soul inline 为主 + `correct_text` 卡片为辅** | 流式体验好；需结构化时才付一次工具调用 |
| TTS | **REST 路由，非 `@tool`** | LLM 无需决策朗读；前端直调更快、可缓存 |
| 水平 | **请求参数传入，不落库** | 避免为 MVP 先建整套 `db/`；落库归二期 |
| 跨会话错误记忆 | **二期** | 依赖 `core/memory/` long 层（当前未落地） |
| persona | **两个独立预置 `AgentConfig`** | 纠错 checklist / 标注更聚焦，契合多 persona 卖点 |
| TTS 依赖 | **edge-tts** | 免费、无需 Key、英/日音色齐；离线/合规需求时可换本地引擎（二期） |

## 17. 追加设计：WebRTC 语音链路优化（2026-06-03）

### 17.1 可行性结论

**可行，但不替代 MVP 的 REST TTS。**

当前 MVP 的「句子旁一键朗读」是短文本、按需播放、可缓存场景，`POST /api/tts` / `/api/tts/stream` 足够简单稳定；为这个场景引入 WebRTC 收益有限，反而会增加信令、PeerConnection、浏览器权限、ICE/TURN 等复杂度。

WebRTC 更适合新增「实时语音陪练」模式：

```text
浏览器麦克风
  -> WebRTC upstream audio track
  -> 后端 STT / VAD / turn detection
  -> language teacher agent
  -> TTS 流式合成
  -> WebRTC downstream audio track
  -> 浏览器低延迟播放
```

因此设计决策：

| 场景 | 协议 | 理由 |
|---|---|---|
| 目标语句子朗读按钮 | REST / StreamingResponse | 简单、可缓存、无需麦克风权限 |
| 聊天文本流式回复 | 现有 WebSocket | 已承载 token、tool.result、Live2D 事件 |
| 实时语音陪练 | WebRTC | 低延迟双向音频、抗抖动、浏览器原生音频管线 |

### 17.2 MVP 调整

MVP 不强制接入 WebRTC，只做以下预留：

- `LanguageLearningView.vue` 保留「语音模式」入口位置，但默认禁用或标记为二期。
- TTS REST 接口继续作为所有朗读按钮的默认实现。
- 后端语言教学工具和 persona 不绑定具体语音协议。

### 17.4 WebRTC 信令接口草案

```http
POST /api/voice/webrtc/offer
Content-Type: application/json
```

请求：

```json
{
  "conversation_id": "web-abc123",
  "persona_id": "japanese_teacher",
  "level": "N5",
  "sdp": "...",
  "type": "offer"
}
```

响应：

```json
{
  "sdp": "...",
  "type": "answer"
}
```

数据通道事件：

```json
{"type": "transcript.partial", "text": "すみません"}
{"type": "transcript.final", "text": "すみません、寿司をください"}
{"type": "agent.text", "text": "这里助词用得不错..."}
{"type": "voice.error", "content": "STT failed"}
```

### 17.5 风险与前置条件

- WebRTC 只解决音频传输低延迟，不直接提供 STT；实时语音陪练仍需要 STT、VAD 和轮次切分。
- 公网部署通常需要 STUN/TURN；本地开发可先只用 host candidate。
- `edge-tts` 是否能稳定提供足够细粒度的低延迟 chunk 需要实测；如果延迟不可接受，二期再评估实时 TTS provider。
- WebRTC 音频链路测试复杂度明显高于 REST TTS，需要补充浏览器端手动验收与后端 pipeline 单元测试。
