# 实时语音链路设计

> 本文是对当前 `WS /ws/{conversation_id}` 文本流式聊天、`POST /api/tts` 文本转语音链路的增量升级设计。
> 当前用户需求写作 "webRTS"，本文按实时语音标准协议 **WebRTC** 设计；若后续确认是其他私有协议，需替换本文的信令与媒体传输章节。
>
> 设计阶段只写文档，不动实现代码；确认后再切开发分支实现。

---

## 1. 目标与范围

### 1.1 功能 1：用户语音输入

用户可以在聊天输入区点击麦克风，说话后由系统识别为文本，并以一次用户消息进入现有 Agent 图：

```
用户麦克风
  → 前端采集音频
  → 后端 ASR 转写
  → 生成 user message
  → build_agent(...).astream_events(...)
  → Agent 回复
```

前端体验要求：
- 麦克风按钮支持 `idle / connecting / listening / transcribing / error` 状态。
- 识别中展示临时转写文本，最终转写进入聊天记录。
- 支持手动停止、取消本轮语音、回退为文本输入。
- 浏览器无麦克风权限、ASR 失败、实时链路断开时，不影响原文本输入。

后端要求：
- ASR provider 走抽象接口，不把具体供应商逻辑写进 route。
- 测试中使用 fake ASR，不发真实语音识别请求。
- 转写后的文本继续复用现有 LangGraph Agent 调用链，避免为语音重写业务逻辑。

### 1.2 功能 2：升级为实时语音链路

把当前链路：

```
WebSocket 文本 token
  → 前端累积文本
  → 分句调用 POST /api/tts
  → 播放 MP3 Blob
```

升级为：

```
WebRTC
  ├── 上行音频 track：用户麦克风 Opus/RTP
  ├── 下行音频 track：助手语音 Opus/RTP
  └── RTCDataChannel：转写、文本 token、工具结果、Live2D 表情、错误事件
```

升级后 `/ws/{conversation_id}` 不再作为实时聊天主链路；保留为兼容文本调试入口，前端主聊天页默认使用 WebRTC。

## 2. 当前链路基线

### 2.1 后端

当前相关文件：
- `asuka/routes/ws.py`：WebSocket 接收 `{message}`，调用 `agent.astream_events(...)`，下发 `token / tool.result / live2d.emotion / done / error`。
- `asuka/routes/tts.py`：`POST /api/tts` 返回完整 MP3，`POST /api/tts/stream` 返回 MP3 chunk。
- `asuka/api/provider/tts.py`：基于 `edge-tts` 合成语音。
- `asuka/core/graph/dispatch.py`：构建 Agent，仍是语音链路应复用的核心入口。

### 2.2 前端

当前相关文件：
- `frontend/src/composables/useChatSocket.ts`：连接 `/ws/{conversation_id}`，处理 token、工具结果、Live2D 事件。
- `frontend/src/composables/useTtsAudio.ts`：按文本快照切句，调用 `/api/tts`，用 Web Audio 分析振幅驱动口型。
- `frontend/src/components/chat/ChatInput.vue`：只有文本输入和发送按钮。
- `frontend/src/types/chat.ts`：定义当前 WebSocket 事件类型。

### 2.3 当前问题

| 问题 | 原因 | 影响 |
|---|---|---|
| 用户不能直接说话 | 无麦克风采集与 ASR | 只能键盘输入 |
| TTS 首响慢 | 先等文本分句，再请求 `/api/tts` | 助手回复的语音响应不够实时 |
| 语音与表情同步弱 | 表情事件来自文本流，音频另一路 HTTP 播放 | Live2D 表情、口型与声音存在漂移 |
| WebSocket 不适合音频媒体 | WS 是消息通道，不处理抖动、丢包、编解码、回声消除 | 实时语音体验上限低 |

## 3. 总体架构

### 3.1 升级后调用链

```
ChatView
  → useRealtimeVoice()
      → getUserMedia({ audio: true })
      → RTCPeerConnection
      → HTTP signaling: offer/answer/ice
      → RTCDataChannel("asuka-events")
          Client → Server:
            text.send / input.audio.commit / chat.cancel / session.update
          Server → Client:
            input.transcript.partial / input.transcript.final
            chat.token / tool.result / live2d.emotion / voice.speech_start / voice.speech_end / done / error
      → remote audio track playback

Backend
  → routes/realtime.py              HTTP 信令入口
  → core/voice/session.py           RealtimeVoiceSession 生命周期
  → core/voice/asr.py               ASR 抽象与分段转写
  → core/voice/tts.py               实时 TTS 抽象，复用 edge-tts provider 起步
  → core/voice/audio.py             音频帧、VAD、队列、下行 track
  → core/graph/dispatch.py          复用现有 Agent 调用
```

### 3.2 传输职责划分

| 通道 | 内容 | 说明 |
|---|---|---|
| WebRTC audio uplink | 用户麦克风音频 | 浏览器编码 Opus，后端接收 RTP 音频帧 |
| WebRTC audio downlink | 助手合成语音 | 后端把 TTS 音频转为下行音频 track |
| RTCDataChannel | 文本、状态、Live2D、工具事件 | 替代当前 WebSocket 事件流 |
| HTTP REST | WebRTC 信令 | 创建 session、交换 offer/answer、补充 ICE candidate |

不使用 WebSocket 做实时语音主链路。信令用 HTTP 是为了满足“用 WebRTC 取代 WebSocket”的边界，同时保持 FastAPI route 简单可测。

## 9. 追加设计：Volcengine V3 WebSocket 单向流式 TTS

> 追加日期：2026-06-04
> 背景：当前实时语音输出存在句间延迟明显、偶现重复播放同一句的问题。用户指定参考火山引擎文档 `https://www.volcengine.com/docs/6561/1719100?lang=zh`，要求切换到 WebSocket 单向流式 V3。

### 9.1 当前问题判断

当前 `asuka/core/voice/tts.py` 中已有 `VolcengineTtsProvider`，但实现走的是 V3 HTTP 单向接口：

```
https://openspeech.bytedance.com/api/v3/tts/unidirectional
```

该 provider 的契约是：

```
synthesize(text) -> SynthesizedPcm
```

也就是必须等服务端响应体收完、解码完成后，`RealtimeVoiceSession` 才能把整句 PCM 放进 `QueuedAudioTrack`。这会造成：

- 句间首包延迟：每个分句要等完整请求结束，下一句才能稳定播放。
- 播放粒度过粗：WebRTC 下行 track 不能边合成边播放，只能播放整句音频。
- 重复播放风险：如果 TTS task 与 turn cancel / done / 后续任务交错，旧任务可能再次 enqueue 已合成的句子。

之前已用 turn 内 TTS 序号修复“后句先完成导致乱序”的问题，但这只能保证顺序，不能解决“整句等待”和“任务生命周期”导致的体验问题。

### 9.2 官方接口选择

按火山引擎豆包语音 API 列表，V3 单向 WebSocket 流式接口用于“一次性输入合成文本，流式输出音频”，地址为：

```
wss://openspeech.bytedance.com/api/v3/tts/unidirectional/stream
```

同一 API 列表中，V1 WebSocket 单向流式接口被标记为不推荐；大模型音色推荐使用 V3 接口，时延表现更好。当前实现不应退回 V1。

### 9.3 目标链路

把当前链路：

```
LLM token
  -> sentence_segments()
  -> Volcengine HTTP request
  -> 收完整句音频
  -> enqueue_pcm()
```

调整为：

```
LLM token
  -> sentence_segments()
  -> Volcengine V3 WebSocket request
  -> async for PCM chunk
  -> 按 turn/sequence 校验
  -> chunk 到达即 enqueue_pcm()
```

外层仍保留 WebRTC 下行音频，不让浏览器直接连火山引擎。原因：

- API Key 仍只存在后端环境变量。
- 前端不新增供应商协议耦合。
- 现有 Live2D 口型、DataChannel 状态事件、ASR/Agent 编排继续复用。

### 9.4 后端接口设计

在 `asuka/core/voice/tts.py` 中扩展 provider 能力：

```python
class RealtimeTtsProvider(Protocol):
    async def synthesize(self, text: str, *, language: str | None) -> SynthesizedPcm: ...

class StreamingRealtimeTtsProvider(RealtimeTtsProvider, Protocol):
    async def stream_synthesize(
        self,
        text: str,
        *,
        language: str | None,
    ) -> AsyncIterator[SynthesizedPcm]: ...
```

兼容策略：

- 支持 `stream_synthesize()` 的 provider 走 chunk 流式播放。
- 只支持 `synthesize()` 的 provider 保持旧行为，用于 Piper / Edge / Qwen3 fallback。
- 测试 fake provider 可实现 `stream_synthesize()`，模拟后一句先返回、取消中断、空 chunk、错误 chunk。

### 9.5 Volcengine provider 设计

新增或改造 `VolcengineTtsProvider`：

- 默认 base URL 改为 `wss://openspeech.bytedance.com/api/v3/tts/unidirectional/stream`。
- 音频格式默认改为 `pcm`，避免流式 wav 多次返回 header 后再额外剥离。
- 认证 header 复用现有 `X-Api-Key` 或 `X-Api-App-Id` / `X-Api-App-Key` / `X-Api-Access-Key` 分支。
- 请求 body 继续使用现有结构：

```json
{
  "user": {"uid": "asuka-realtime"},
  "req_params": {
    "text": "...",
    "speaker": "...",
    "audio_params": {
      "format": "pcm",
      "sample_rate": 24000
    }
  }
}
```

实现约束：

- 使用 `websockets` 异步客户端，项目 lock 中已有该依赖。
- 每次分句创建一个 WebSocket 单向请求；后续若要进一步降低 LLM 分句等待，可再评估 V3 双向流式。
- provider 只负责把服务端二进制或 JSON/SSE 风格消息解析为 `SynthesizedPcm` chunk，不直接操作 WebRTC track。
- `TtsProviderError` 中包含 request id，方便排查供应商错误。

### 9.6 Session 播放顺序与去重设计

`RealtimeVoiceSession` 中的 `_schedule_tts()` 继续给每个分句分配递增 `sequence`。改造 `_synthesize_sentence()`：

1. 合成任务开始后先调用 provider。
2. 如果 provider 支持流式，收到首个 chunk 前等待自己的 `sequence` 播放权。
3. 获得播放权后发送一次 `voice.speech_start`。
4. 每个 chunk 到达立即 `outbound_audio.enqueue_pcm(...)`。
5. 流结束后发送一次 `voice.speech_end`，再推进 `_next_tts_enqueue_sequence`。

去重和取消规则：

- 每个 turn 开始时重置 TTS sequence、skipped set、active audio generation id。
- 每个 TTS task enqueue 前必须检查 `active_turn_id == turn_id`。
- 取消或新 turn 开始时 cancel 所有 TTS task，并清空 outbound queue。
- 同一句只允许发送一次 `voice.speech_start` / `voice.speech_end`。
- 如果某个 sequence 失败，调用 `_skip_tts_sequence(sequence)`，让后续句子继续播放，不重复失败句。

### 9.9 预期效果与边界

预期改善：

- 首个音频 chunk 到达后立即进入 WebRTC 下行队列，降低句间空白。
- TTS task 生命周期更严格，减少旧任务重复 enqueue 造成的重复播放。
- 使用 V3 WebSocket 单向流式，避免当前 HTTP 整句等待。

边界：

- 该方案仍按句触发 TTS；如果 LLM 一直不输出句末标点，仍要等 `force=True` 或分句策略触发。
- 若要进一步降低“首句之前”的等待，需要改 `sentence_segments()`，支持逗号、短语长度阈值或 V3 双向流式文本输入。
- WebSocket V3 的二进制协议细节需以官方文档和实际返回为准，测试中必须覆盖已观察到的消息格式。

### 9.10 变更决策：改用 V3 WebSocket 双向流式

> 追加日期：2026-06-04
> 用户确认从单向流式升级为双向流式。

对实时 Agent 语音交互，V3 双向流式比 V3 单向流式更合适：

- 单向流式：一次性输入一段文本，服务端流式输出音频。它可以降低“整句音频响应体等待”，但仍然需要先切出完整句子后才能请求 TTS。
- 双向流式：客户端在同一个 WebSocket session 内持续输入文本片段，服务端持续输出音频。它更适合 LLM token 流式生成，可减少句间等待和重复建连开销。

新目标链路：

```
LLM token
  -> 清理 Live2D / tool JSON 后的可见文本
  -> TTS text buffer
  -> 按标点或最小长度切成可播报片段
  -> Volcengine V3 bidirection TaskRequest
  -> TTSResponse audio chunks
  -> WebRTC outbound_audio.enqueue_pcm()
```

实现策略：

- `VolcengineTtsProvider.stream_synthesize()` 使用 `wss://openspeech.bytedance.com/api/v3/tts/bidirection`。
- 当前第一版仍以“片段”为单位调用 bidirectional session，以保证稳定交付；片段切分从强句号放宽到“标点或长度阈值”，降低句间等待。
- 后续优化可以把一个 Agent turn 绑定到一个长期 bidirectional connection，LLM token 进入同一个 `TaskRequest` 序列。
- session 层继续保留 `turn_id + sequence` 校验，旧 turn、已取消 task、失败 sequence 均不能再入队。
- 音频格式默认使用 `pcm`，避免流式 `wav` 多 header 问题。

### 4.4 DataChannel 事件协议

DataChannel 名称：`asuka-events`

所有事件统一格式：

```ts
interface RealtimeEvent<T = unknown> {
  type: string
  id: string
  conversationId: string
  sessionId: string
  turnId?: string
  createdAt: number
  payload: T
}
```

Client → Server：

| type | payload | 说明 |
|---|---|---|
| `session.update` | `{ personaId?, level?, language?, voice? }` | 更新本会话配置 |
| `text.send` | `{ content: string }` | 文本兜底输入，走同一实时链路 |
| `input.audio.commit` | `{}` | 手动提交当前语音段，绕过 VAD 等待 |
| `chat.cancel` | `{ turnId?: string }` | 取消当前 LLM/TTS 输出 |
| `ping` | `{ ts: number }` | 延迟检测 |

Server → Client：

| type | payload | 说明 |
|---|---|---|
| `session.ready` | `{}` | DataChannel 可用 |
| `input.speech_start` | `{}` | VAD 检测到用户开始说话 |
| `input.speech_end` | `{}` | VAD 检测到用户停止说话 |
| `input.transcript.partial` | `{ text: string }` | 临时转写 |
| `input.transcript.final` | `{ text: string }` | 最终转写，前端写入用户消息 |
| `chat.token` | `{ content: string }` | Agent 文本 token，替代 WS `token` |
| `tool.result` | `{ name: string, payload: unknown }` | 复用现有工具结果语义 |
| `live2d.emotion` | `{ emotion?, motion?, expression?, durationMs?, intensity? }` | 复用现有 Live2D 事件 |
| `voice.speech_start` | `{ turnId: string }` | 助手下行语音开始播放 |
| `voice.speech_end` | `{ turnId: string }` | 助手下行语音结束 |
| `done` | `{ turnId: string }` | 一轮回复结束 |
| `error` | `{ message: string, code?: string }` | 错误 |
| `pong` | `{ ts: number }` | 延迟检测返回 |

### 4.5 会话核心对象

```python
class RealtimeVoiceSession:
    session_id: str
    conversation_id: str
    agent_config: AgentConfig
    peer_connection: RTCPeerConnection | None
    data_channel: RTCDataChannel | None
    inbound_audio_queue: asyncio.Queue[AudioFrame]
    outbound_audio_track: MediaStreamTrack
    active_turn_id: str | None

    async def attach_offer(self, offer: RTCSessionDescription) -> RTCSessionDescription: ...
    async def handle_audio_frame(self, frame: AudioFrame) -> None: ...
    async def handle_text(self, content: str) -> None: ...
    async def run_agent_turn(self, user_text: str) -> None: ...
    async def close(self) -> None: ...
```

核心原则：
- `routes/realtime.py` 只做 HTTP → core 映射，不写业务逻辑。
- `run_agent_turn()` 复用 `build_agent()` 和 `astream_events()`，沿用 `conversation_id` checkpoint。
- Live2D 标签清理逻辑应从 `routes/ws.py` 抽到共享模块，避免实时链路和 WS 链路复制正则。
- 工具结果 `_jsonable_tool_payload()`、语言工具 JSON 清理也应抽成共享 helper。

### 4.6 ASR 设计

Provider 接口：

```python
class ASRProvider(Protocol):
    async def transcribe(self, audio: AudioSegment, language: str | None = None) -> str: ...
```

MVP 流程：

```
WebRTC audio frames
  → audio.py 采样/重采样为 16k mono PCM
  → VAD 收集一句话
  → asr.py 调 ASRProvider.transcribe()
  → DataChannel: input.transcript.final
  → session.run_agent_turn(text)
```

VAD 初始策略：
- 音量阈值 + 静音持续时间，先做轻量实现。
- 后续可替换为 `webrtcvad` 或 Silero VAD。

边界：
- 一段语音最长 20 秒，超过自动 commit。
- 转写为空不触发 Agent。
- ASR 失败只发 `error`，不结束 WebRTC session。

### 4.7 TTS 设计

Provider 接口：

```python
class RealtimeTTSProvider(Protocol):
    async def synthesize_chunks(
        self,
        text: str,
        voice: str | None = None,
        language: str | None = None,
    ) -> AsyncIterator[bytes]: ...
```

MVP 先复用现有 `edge-tts`：
- LLM token 仍按标点和长度切成短句。
- 每个短句调用 `iter_speech_chunks()`。
- 后端将 MP3 chunk 解码/转 PCM 后写入 outbound audio track。

后续低延迟优化：
- 接入原生实时 TTS provider，直接输出 PCM/Opus。
- 对第一句使用更小阈值，后续句子并发预合成。
- DataChannel `voice.speech_start/end` 与下行 audio track 同步，Live2D 口型用远端音频 `AnalyserNode` 驱动。

## 10. 后端本地 ASR/TTS 链路补充

本节补充从信令 MVP 升级到完整后端实时语音链路的实现约束。实时模式保留 `RTCDataChannel` 作为文本和事件通道，新增 WebRTC 上行音频到后端 ASR、后端 TTS 到 WebRTC 下行音频。

### 10.1 Provider 与配置

- ASR provider 使用 `faster-whisper`，默认模型 `small`，模型名、设备和 compute type 从 `Settings` 读取。
- TTS provider 使用 Piper，本地模型路径通过 `piper_model_by_language` 按语言映射，不提交模型文件。
- 新增配置：`asr_model`、`asr_device`、`asr_compute_type`、`voice_vad_rms_threshold`、`voice_vad_min_speech_ms`、`voice_vad_silence_ms`、`voice_vad_max_speech_ms`、`piper_model_by_language`、`piper_sample_rate`。
- Provider 初始化延迟执行，测试可注入 fake provider，避免真实模型下载或合成。

### 10.2 上行音频与 ASR

WebRTC 上行 `AudioFrame` 在 session 内被重采样为 16kHz mono s16 PCM。RMS VAD 使用配置阈值检测：

- 从静音进入语音时发送 `input.speech_start`。
- 静音达到 `voice_vad_silence_ms` 或收到 `input.audio.commit` 时提交当前语音段。
- 单段超过 `voice_vad_max_speech_ms` 时强制提交。
- 有效语音段短于 `voice_vad_min_speech_ms` 时丢弃，不触发 Agent。
- `faster-whisper` v1 不发送 partial transcript，只发送 `input.transcript.final`。

ASR 结果为空时只清理状态，不触发 Agent。`text.send` 不经过 ASR，仍直接进入 Agent turn。

### 10.3 下行音频与 TTS

下行 track 替换为 `QueuedAudioTrack`，固定输出 48kHz mono s16 PCM。无语音时持续输出静音；TTS 写入 PCM 队列后按 20ms frame 出队。

Agent turn 中继续通过 DataChannel 推送 `chat.token`、`tool.result`、`live2d.emotion` 和 `done`。同时把清洗后的可见文本按中英日句末标点切句，逐句调用 Piper provider 合成，转换为 48kHz mono PCM 后写入下行队列。每个有音频的句段通过 DataChannel 发送 `voice.speech_start` 和 `voice.speech_end`，供前端同步说话状态。

Piper 缺少对应语言模型时发送 `error` 事件，但不中断文本 token 输出。

### 10.4 取消与兼容

`chat.cancel` 必须取消当前 Agent task，清空 VAD/ASR 缓冲和 TTS 下行队列，停止后续语音输出。旧 `/ws` 与 `/api/tts` 保留为兼容和调试入口；实时模式前端不再调用浏览器 SpeechRecognition 或 `/api/tts`。

### 10.5 Qwen3-TTS Provider

实时 TTS provider 支持通过 `REALTIME_TTS_PROVIDER` 切换：

- `piper`：默认，本地 Piper `.onnx` voice model。
- `qwen3`：DashScope Qwen3-TTS HTTP provider，默认模型 `qwen3-tts-flash`。

Qwen3-TTS 通过 `DASHSCOPE_API_KEY` 鉴权，按句请求非流式接口。provider 从响应中的 `output.audio.url` 下载 WAV；若响应包含流式同款 `output.audio.data` Base64 音频，也会直接解码。最终统一转换为 mono s16 PCM，复用现有 `QueuedAudioTrack` 下行播放。

推荐配置：

```env
REALTIME_TTS_PROVIDER=qwen3
DASHSCOPE_API_KEY=sk-...
QWEN_TTS_MODEL=qwen3-tts-flash
QWEN_TTS_VOICE=Cherry
```

语言映射沿用应用层 `chinese / english / japanese`，默认映射到 Qwen3-TTS 的 `Chinese / English / Japanese`。

## 11. 火山引擎实时语音 API 接入设计

本节追加火山引擎语音合成能力接入方案。目标是在不改变前端 WebRTC 主链路的前提下，把后端 TTS provider 切换为火山云端能力；ASR 继续使用本地 `faster-whisper`。

```
Browser WebRTC
  ├── 上行 audio track → Asuka 后端 VAD/PCM
  ├── 下行 audio track ← Asuka 后端 PCM queue
  └── DataChannel 文本/事件

Asuka 后端
  ├── FasterWhisperAsrProvider：PCM segment → 本地 faster-whisper → transcript
  ├── LangGraph Agent：transcript/text.send → token
  └── Volcengine TTS provider：sentence → 火山语音合成 → PCM queue
```

### 11.1 选型边界

- v1 只接入火山 OpenSpeech/豆包语音合成 API，作为 `RealtimeTtsProvider` 的云端实现。
- 不把浏览器直接接到火山实时音视频 RTC 房间。当前项目需要把音频先进 Asuka 后端，才能复用 VAD、Agent、DataChannel 事件、Live2D 表情和下行音频队列。
- TTS 优先接火山 HTTP 单向流式 V3 接口；ASR 不接火山，继续复用本地 `faster-whisper`。
- 火山 API Key、App ID、Access Key、Resource ID 等只从环境变量读取，不写入代码或测试。

官方接口路线按用途区分：

- 豆包语音 TTS 大模型推荐 V3 接口。可选接口包括 `wss://openspeech.bytedance.com/api/v3/tts/bidirection`、`wss://openspeech.bytedance.com/api/v3/tts/unidirectional/stream`、`https://openspeech.bytedance.com/api/v3/tts/unidirectional` 和 `https://openspeech.bytedance.com/api/v3/tts/unidirectional/sse`。
- 豆包语音 ASR 支持大模型流式语音识别，但本阶段明确不接入。
- 火山边缘智能 Realtime API 的地址是 `wss://ai-gateway.vei.volces.com/v1/realtime?model=<model>`，用于调用自部署 ASR/TTS 模型。本阶段不接这条路线。

### 11.2 配置项

新增配置建议：

```env
REALTIME_TTS_PROVIDER=volcengine

VOLCENGINE_TTS_API_KEY=
VOLCENGINE_TTS_APP_ID=
VOLCENGINE_TTS_ACCESS_KEY=
VOLCENGINE_TTS_RESOURCE_ID=
VOLCENGINE_TTS_VOICE_TYPE=
VOLCENGINE_TTS_ENCODING=wav
VOLCENGINE_TTS_SAMPLE_RATE=24000
VOLCENGINE_API_TIMEOUT_SECONDS=30
```

`REALTIME_TTS_PROVIDER` 支持：

- `piper`：默认本地 TTS。
- `qwen3`：DashScope Qwen3-TTS。
- `volcengine`：火山 TTS。

### 11.3 后端模块

新增或扩展：

```
asuka/core/voice/
├── asr.py                       # 不新增火山 ASR，继续 FasterWhisperAsrProvider
├── tts.py
│   ├── PiperTtsProvider
│   ├── Qwen3TtsProvider
│   └── VolcengineTtsProvider
```

`VolcengineTtsProvider` 第一版使用 HTTP 单向流式响应，不新增 WebSocket 依赖。火山返回的 base64 音频 chunk 在 provider 内聚合并转换成 mono s16 PCM。

### 11.4 ASR 行为

ASR 明确继续沿用当前后端 VAD + 本地 faster-whisper：

```
WebRTC audio frame
  → 16k mono s16 PCM
  → RmsVadSegmenter
  → FasterWhisperAsrProvider.transcribe(segment)
  → input.transcript.final
  → start_turn(text)
```

本阶段不新增 `REALTIME_ASR_PROVIDER`，也不发送 ASR partial transcript。

### 11.5 TTS 行为

Agent token 仍按句切分：

```
chat.token buffer
  → sentence_segments()
  → VolcengineTtsProvider.synthesize(sentence)
  → SynthesizedPcm
  → QueuedAudioTrack.enqueue_pcm()
```

火山返回音频后统一转换为 mono s16 PCM。若返回采样率不是 `piper_sample_rate` / WebRTC 下行采样率，继续复用 `resample_pcm16_mono()` 转为 48kHz mono s16。

TTS 异常只发送：

```json
{"type": "error", "payload": {"code": "tts_failed", "message": "..."}}
```

文本 token、工具事件、Live2D 事件继续正常输出。

### 11.6 取消与资源释放

`chat.cancel` 除当前行为外，还必须关闭火山侧连接：

- 取消当前 ASR/TTS task。
- 关闭未完成的 WebSocket/HTTP streaming response。
- 清空 `QueuedAudioTrack`。
- 发送 `done(cancelled=true)`。

后端 session close 时也必须释放火山连接，避免云端计费持续。

## 12. Edge TTS 实时 Provider 回退

火山语音合成实测效果不符合预期时，实时 WebRTC 语音链路允许回退到现有 `edge-tts` 能力。旧 `/api/tts` 已经通过 `asuka/api/provider/tts.py` 封装 Edge TTS，实时 provider 不重新实现请求逻辑，只复用旧封装并增加 MP3 到 PCM 的转换：

```
Agent sentence
  → EdgeTtsProvider.synthesize()
  → asuka.api.provider.tts.synthesize_speech()
  → MP3 bytes
  → PyAV decode + mono s16 PCM
  → QueuedAudioTrack.enqueue_pcm()
```

配置方式：

```env
REALTIME_TTS_PROVIDER=edge
TTS_VOICE=zh-CN-XiaoxiaoNeural
```

`REALTIME_TTS_PROVIDER` 支持项追加：

- `edge` / `edge_tts`：微软 Edge TTS，复用现有 `tts_voice`、`tts_voices`、`tts_rate`、`tts_volume`、`tts_pitch` 配置。

Edge TTS 仍是云端公共端点，优点是音质和可用音色更接近之前 `/api/tts` 的体验；缺点是返回 MP3，需要后端额外解码，首响和稳定性取决于 Edge TTS 网络状态。

## 13. 火山 V3 新版控制台 Request Schema 调整

根据 `docs/apiref/火山引擎流式语音合成v3.md` 清洗后的协议参考，当前 `VolcengineTtsProvider` 需要从旧控制台兼容形态收敛到新版控制台默认 schema。

### 13.1 Header 调整

新版控制台默认只需要：

```python
{
    "X-Api-Key": "<VOLCENGINE_TTS_API_KEY>",
    "X-Api-Resource-Id": "seed-tts-2.0",
    "X-Api-Connect-Id": "<uuid>"
}
```

实现约束：

- `VOLCENGINE_TTS_API_KEY` 存在时优先使用新版控制台鉴权。
- `VOLCENGINE_TTS_RESOURCE_ID` 默认改为 `seed-tts-2.0`，不再使用旧 `volc.service_type.10029`。
- `X-Api-Connect-Id` 替代当前内部使用的 `X-Api-Request-Id`，用于连接追踪。
- 旧版控制台仅作为兼容路径：`X-Api-App-Id + X-Api-Access-Key + X-Api-Resource-Id + X-Api-Connect-Id`。
- 旧版兼容路径不再发送 `X-Api-App-Key`，因为 V3 双向流式文档未要求该字段。

### 13.2 Payload 调整

双向流式 WebSocket 的 JSON payload 统一包含：

```json
{
  "event": 100,
  "namespace": "BidirectionalTTS",
  "user": {
    "uid": "asuka-realtime"
  },
  "req_params": {
    "speaker": "<voice_type>",
    "audio_params": {
      "format": "pcm",
      "sample_rate": 24000
    }
  }
}
```

`TaskRequest` 使用相同顶层 schema，并在 `req_params.text` 中传入待合成文本：

```json
{
  "event": 200,
  "namespace": "BidirectionalTTS",
  "user": {
    "uid": "asuka-realtime"
  },
  "req_params": {
    "text": "待合成文本"
  }
}
```

实现约束：

- `StartSession` 只传会话级配置，不传空字符串 `text`。
- `TaskRequest` 只传当前文本片段和必要扩展参数，避免重复发送音色配置造成协议语义混乱。
- `explicit_language` 继续放在 `req_params.additions`，仅在语言可映射且确有必要时发送；中文默认不发送，避免部分 TTS 2.0 音色在任务阶段无响应。
- 保留 `_request_payload()` 仅作为测试和兼容辅助，返回与 `TaskRequest` 一致的新版 schema。

### 13.3 Smoke Test 结论

2026-06-04 本地 smoke test 结果：

- `seed-tts-2.0 + zh_female_sajiaonvyou_moon_bigtts`：建连成功、`SessionStarted` 成功，但 `TaskRequest` 后返回 `55000000`，判断为 1.0/2.0 音色与 Resource ID 不匹配。
- `seed-tts-2.0 + zh_female_vv_uranus_bigtts`：建连成功、`SessionStarted` 成功，`TaskRequest` 后收到 `TTS_RESPONSE` 音频帧。
- `seed-tts-2.0 + zh_female_vv_uranus_bigtts + explicit_language=zh-cn`：provider 层等待音频超时；中文默认不传 `explicit_language` 后可收到 PCM chunk。

因此默认配置调整为：

```env
VOLCENGINE_TTS_RESOURCE_ID=seed-tts-2.0
VOLCENGINE_TTS_VOICE_TYPE=zh_female_vv_uranus_bigtts
```

### 13.4 测试调整

新增或更新测试断言：

- 新版 API Key header 不包含 `X-Api-Request-Id`，包含 `X-Api-Connect-Id`。
- 默认 `VOLCENGINE_TTS_RESOURCE_ID=seed-tts-2.0`。
- 旧版兼容 header 不包含 `X-Api-App-Key`。
- `StartSession` payload 包含 `event=100`、`namespace=BidirectionalTTS`、`user` 和会话级 `req_params`。
- `TaskRequest` payload 包含 `event=200`、`namespace=BidirectionalTTS`、`user` 和 `req_params.text`。
